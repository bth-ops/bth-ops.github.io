Låt oss kika på vad en lagrad funktion är. Det handlar om `CREATE FUNCTION` och "compound statements" och hur det används tillsammans med SQL-frågor. 

En kort förklaring är att det är en möjlighet att skriva små program i databasen som kan användas i SQL-satser, t.ex. i SELECT-satser. Det är ett sätt att göra SQL-koden mer kraftfull och mer läsbar, eller att göra komplexa frågor på ett strukturerat sätt genom att dela upp kod.

## Lagrade funktioner

Idag skall vi lösa detta problemet med något som heter lagrade funktioner. Det vi vill ha är en funktion som tar en poäng som inparameter och returnerar ett betyg.

Tanken är att skriva en funktion som kan användas i en SELECT-sats och returnera betyget utifrån antalet poäng. Funktionen skall alltså översätta ett poäng till ett betyg, inuti en SELECT-sats.

Så här ska det kunna se ut.

```sql title="Exempel på hur en lagrad funktion kan användas i en SELECT-sats."
SELECT
    acronym,
    score,
    grade(score) AS grade
FROM exam;
```

Det är alltså delen med `grade(score)` som är anropet till en lagrad funktion.

Kunde man inte gjort detta med en tabell som innehåller betygstabellen och joinat med ett smart villkor eller använd en sub-query? Jo, det kan man säkert. Men nu använder vi funktioner. I databaser finns ofta olika varianter på att lösa ett specifikt problem.



## CREATE FUNCTION

Vi börjar med att skapa en enkel funktion för att lära oss konceptet och hur det används.

Jag skapar en funktion som bara returnerar inkommande parameter. Det är bara för att ha ett skal, en mall, att komma igång med.

Här är mallen till en sådan funktion.

```sql
--
-- Function for grading an exam.
--
DROP FUNCTION IF EXISTS `grade`;
DELIMITER ;;

CREATE FUNCTION `grade`(
    p_score INTEGER
)
RETURNS INTEGER
DETERMINISTIC
BEGIN
    RETURN p_score + 1;
END
;;

DELIMITER ;
```

Funktionen tar en parameter och returnerar ett värde. För tillfället returneras samma värde vi skickar in i funktionen, men med +1.

Som du ser så används DELIMITER till att avgränsa koden så att semikolon fungerar som avslutare, inne i funktionens body.

En funktion ser ut som en lagrad procedur, det som skiljer är att funktionen returnerar ett värde via konstruktionen `RETURNS INTEGER`. En lagrad procedur kan inte returnera ett värde på det viset, en lagrad procedure producerar resultset och kan ha parametrar som är IN/OUT.

Funktionen kan användas så här.

```sql
SELECT
    `acronym`,
    `score`,
    grade(score) AS 'grade'
FROM `exam`;
```

Kör man mot tabellen så blir utskriften så här.

```sql
MariaDB [mydb]> SELECT
    ->     `acronym`,
    ->     `score`,
    ->     grade(score) AS 'grade'
    -> FROM `exam`;
+---------+-------+-------+
| acronym | score | grade |
+---------+-------+-------+
| adam    |    77 |    78 |
| john    |    63 |    64 |
| june    |    49 |    50 |
| meta    |    97 |    98 |
| siva    |    88 |    89 |
| ubbe    |    52 |    53 |
+---------+-------+-------+
```

Låt oss göra en smartare funktion som kan ge oss betyget enligt betygsskalan.



## Funktion för betygsättning

I en funktion kan vi skriva compound statements. Det låter som vi kan lösa uppgiften med en IF, ELSEIF och END IF-sats.

Glöm inte att ändra return-typen till `CHAR(2)` så den matchar det som nu returneras.

```sql
--
-- Function for grading an exam A-F, FX.
--
DROP FUNCTION IF EXISTS grade;
DELIMITER ;;

CREATE FUNCTION grade(
    score INTEGER
)
RETURNS CHAR(2)
DETERMINISTIC
BEGIN
    IF score >= 90 THEN
        RETURN 'A';
    ELSEIF score >= 80 THEN
        RETURN 'B';
    ELSEIF score >= 70 THEN
        RETURN 'C';
    ELSEIF score >= 60 THEN
        RETURN 'D';
    ELSEIF score >= 55 THEN
        RETURN 'E';
    ELSEIF score >= 50 THEN
        RETURN 'Fx';
    END IF;
    RETURN 'F';
END
;;

DELIMITER ;
```

Då kan vi provköra SELECT och använda funktionen för att skapa en betygslista.

```sql
SELECT
    `acronym`,
    `score`,
    grade(score) AS 'grade'
FROM `exam`
ORDER BY `score` DESC;
```

Så här kan det se ut.

```sql
MariaDB [mydb]> SELECT
    ->     `acronym`,
    ->     `score`,
    ->     grade(score) AS 'grade'
    -> FROM `exam`
    -> ORDER BY `score` DESC;
+---------+-------+-------+
| acronym | score | grade |
+---------+-------+-------+
| meta    |    97 | A     |
| siva    |    88 | B     |
| adam    |    77 | C     |
| john    |    63 | D     |
| ubbe    |    52 | Fx    |
| june    |    49 | F     |
+---------+-------+-------+
```

Fint, det verkar som det fungerar som tänkt. Så använder man alltså en lagrad funktion i en SELECT-sats. Det är ett kraftfullt verktyg som kan göra SQL-koden mer läsbar och mer strukturerad, eller som ett sätt att göra komplexa frågor mer hanterbara genom att dela upp kod i mindre delar.

Om du läser på om compound statements så ser du att man även kan göra en SWITCH/CASE-sats, det är ett alternativ till IF/ELSEIF/END IF. Det kan vara mer läsbart i vissa fall. Testa gärna att skriva om funktionen med SWITCH/CASE istället för IF/ELSEIF.

<details>
<summary>Den lagrade funktionen med SWITCH/CASE.</summary>

Här är funktionen omskriven med SWITCH/CASE.

Först ser vi en variant som returnerar i varje CASE-sats.

```sql
CREATE FUNCTION `grade`(
    p_score INTEGER
)
RETURNS CHAR(2)
DETERMINISTIC
BEGIN
    CASE
        WHEN p_score >= 90 THEN RETURN 'A';
        WHEN p_score >= 80 THEN RETURN 'B';
        WHEN p_score >= 70 THEN RETURN 'C';
        WHEN p_score >= 60 THEN RETURN 'D';
        WHEN p_score >= 55 THEN RETURN 'E';
        WHEN p_score >= 50 THEN RETURN 'Fx';
        ELSE RETURN 'F';
    END CASE;
END
;;
```

Här är en variant där vi använder CASE som ett "statement"

```sql
CREATE FUNCTION `grade`(
    p_score INTEGER
)
RETURNS CHAR(2)
DETERMINISTIC
BEGIN
    RETURN (
        CASE
            WHEN p_score >= 90 THEN 'A'
            WHEN p_score >= 80 THEN 'B'
            WHEN p_score >= 70 THEN 'C'
            WHEN p_score >= 60 THEN 'D'
            WHEN p_score >= 55 THEN 'E'
            WHEN p_score >= 50 THEN 'Fx'
            ELSE 'F'
        END
    );
END
;;
```

Du får själv bedöma om du tycker att någon av dessa varianter blev bättre än den med IF/ELSEIF.

</details>



## Skapa ytterligare en funktion

Läraren blir plötsligt uppmärksammad på att betygsskalan var en annan, för en del av studenterna. En skala som gick mellan 3-5 och U.

Så här ser lärarens uppdaterade betygstabell ut, en kombination mellan betygen A-F och U,3-5.

| A-F | U,3-5 | Poäng | Definition |
|:---:|:-----:|:-----:|------------|
| A   | 5     | 90+   | UTMÄRKT – enastående resultat. |
| B   | 4     | 80+   | MYCKET BRA – klart över medelstandard. |
| C   | 4     | 70+   | BÄTTRE ÄN BRA – allmänt bra arbete. |
| D   | 3     | 60+   | BRA – en fullgod prestation. |
| E   | 3     | 55+   | GODKÄNT – resultatet uppfyller minimikriterierna. |
| FX  | -     | 54-   | KOMPLETTERA – mer arbete krävs innan betyg kan sättas. |
| F   | U     | 50-   | OTILLRÄCKLIGT – (avsevärt) mer arbete krävs. |

Låt oss hjälpa läraren och lösa detta med en ny funktion som ger betyg enligt skalan U, 3-5.

```sql
--
-- Function for grading an exam U, 3-5.
--
DROP FUNCTION IF EXISTS `grade2`;
DELIMITER ;;

CREATE FUNCTION `grade2`(
    p_score INTEGER
)
RETURNS CHAR(1)
DETERMINISTIC
BEGIN
    IF p_score >= 90 THEN
        RETURN '5';
    ELSEIF p_score >= 70 THEN
        RETURN '4';
    ELSEIF p_score >= 55 THEN
        RETURN '3';
    END IF;
    RETURN 'U';
END
;;

DELIMITER ;
```

Egentligen är det samma sak en gång till, nu är det bara andra värden som skall testas och returneras.

Nu kan läraren ta ut sin betygslista så blir det tydligt att se vilket betyg som är aktuellt för varje student.

```sql
SELECT
    `acronym`,
    `score`,
    grade(score) AS 'A-F, FX',
    grade2(score) AS 'U, 3-5'
FROM `exam`
ORDER BY `score` DESC;
```

Så här kan det se ut.

```sql
MariaDB [mydb]> SELECT
    ->     `acronym`,
    ->     `score`,
    ->     grade(score) AS 'A-F, FX',
    ->     grade2(score) AS 'U, 3-5'
    -> FROM `exam`
    -> ORDER BY `score` DESC;
+---------+-------+---------+--------+
| acronym | score | A-F, FX | U, 3-5 |
+---------+-------+---------+--------+
| meta    |    97 | A       | 5      |
| siva    |    88 | B       | 4      |
| adam    |    77 | C       | 4      |
| john    |    63 | D       | 3      |
| ubbe    |    52 | Fx      | U      |
| june    |    49 | F       | U      |
+---------+-------+---------+--------+
```

Vi har fått ett exempel på hur lagrade funktioner kan användas för att presentera informationen på olika sätt och förenkla kodstrukturen kring en annars mer komplex SELECT-sats.



## Karaktäristik av en funktion

I de funktioner vi skapat ovan har vi angivit dem med nyckelordet `DETERMINISTIC`. Det innebär en deklaration av funktionen som säger att den alltid returnerar samma svar givet den parameter som skickas in.

Motsatsen är `NOT DETERMINISTIC` och för att ta ett exempel så har vi en funktion som returnerar nuvarande tid, resultatet som returneras är alltid beroende av andra omständigheter (klockan) än de parametrar som eventuellt skickas in till funktionen. En annan variant av en NOT DETERMINISTIC funktion vore en funktion som beräknar en ålder utifrån ett födelsedatum, det som returneras är inte enbart beroende av parametern födelsedatum.

```sql
--
-- Example of a NOT DETERMINISTIC function.
--
DROP FUNCTION IF EXISTS `time_of_the_day`;
DELIMITER ;;

CREATE FUNCTION `time_of_the_day`()
RETURNS DATETIME
NOT DETERMINISTIC NO SQL
BEGIN
    RETURN NOW();
END
;;

DELIMITER ;
```

Så här ser det ut när funktionen används.

```sql
MariaDB [mydb]> SELECT time_of_the_day();
+---------------------+
| time_of_the_day()   |
+---------------------+
| 2026-02-27 17:08:33 |
+---------------------+
```

I definitionen av funktionen `time_of_the_day()` ser du att `NO SQL` anges. Det är en deklaration som säger att funktionen inte använder några SQL-satser. Motsatsen hade varit deklarationen `CONTAINS SQL`.

Det finns även deklarationer för `READS SQL DATA` och `MODIFIES SQL DATA`.

Dessa deklarationer anges av den som skapar funktionen och databasservern _kan_ använda informationen för att optimera körningarna. Om man deklarerar sin funktion på fel sätt så kan det ge upphov till oväntade resultat.



## SHOW FUNCTION

När man vill se vilka funktioner som finns i databasen så kan man visa dem.

```sql
SHOW FUNCTION STATUS;
SHOW FUNCTION STATUS LIKE 'grade' \G
SHOW FUNCTION STATUS WHERE Db = 'mydb';
```

## Funderingar

Eftersom vi kan skriva compound statements i funktionens body så har vi stora möjligheter att utföra saker i en funktion. Det kan vara enklare utskrifts/rapportstöd, eller kanske en uppslagning i andra tabeller för att hämta värden liknande en subquery.

Man får dock ha i åtanke att en funktion som används i en SELECT-sats kommer att köras för varje rad i resultatet, så det gäller att tänka på prestanda och att inte göra onödigt tunga operationer i en funktion som används i en SELECT-sats. Det är en av nackdelarna med att använda funktioner, det kan ge prestandaproblem om man inte är försiktig. Det är också en av anledningarna till att man inte kan göra vissa saker i en funktion, t.ex. att modifiera data i tabeller, det är inte tillåtet i en funktion som används i en SELECT-sats.

Finns det inte andra sätt att lösa problemet i exemplet? Jo det finns det säkert. Men det är ändå ett rimligt exempel som visar hur du kan ha nytta av lagrade funktioner.

Vi har många verktyg i lådan som används för att lösa problem och utmaningar som uppstår. Man ska bara välja rätt verktyg i rätt situation, det är lite det som är utmaningen.
