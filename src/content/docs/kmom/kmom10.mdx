## Projektidé och upplägg

Du skall fortsätta att bygga ut din eshop med innehåll i databasen och ett tillhörande terminalprogram i CSharp.

Du behöver göra de krav som är grundläggande, de är obligatoriska att genomföra (krav 1-3) och leder ofta fram till betyg D (eller E om man får något avdrag).

Det finns optionella delar som du kan välja att göra för att öka dina chanser till ett högre betyg, (krav 4-6). Normalt kan varje optionellt krav påverka en betygsnivå, det vill säga krav 4 -> C, krav 5 -> B, och krav 6 -> A.

Kraven är byggda för att utföras i den ordning som de är listade.



### Krav 1: CSharp

Utför följande krav.

1. Skapa ett CSharp projekt i katalogen `kmom/10/Eshop` genom att kopiera `kmom/08/Eshop`. 

1. Dela upp koden i klasser. Organisera din kod så att den är DRY. Refaktorisera när det behövs.

1. Samtliga krav från kmom07 och kmom08 skall vara uppfyllda.



### Krav 2: Databasens schema och innehåll


#### Självtest databas

Försäkra dig om att din backupfil fungerar. När läraren återskapar din databas så kommer följande kommando att köras.

```bash
# Gå till katalogen sql/

```

Kontrollera att du även kan återskapa din databas enligt följande.

```bash title="Återskapa databas"
# Gå till katalogen sql/
mariadb < setup.sql
mariadb eshop < proc.sql
mariadb eshop < insert.sql
```

Flödet ovan behöver fungera, om din backupfil inte fungerar så kan läraren välja att prova att återställa din databas från dessa filer. Det är tillåtet att dela upp din kod i fler filer och göra source på dem samt att läsa in data från till exempel csv-filer.

Filerna `proc.sql` och `insert.sql` får inte innehålla ett `USE eshop` för att välja databas. Det gör att läraren får problem vid rättning.

Kontrollera att databasen innehåller tillräckligt med data.



### Krav 3: CRUD product

Implementera följande menyval.

```bash
product_add <title> <base_price>  - Add a new product with its base price
product_edit <product_id>         - Edit details of a product
product_del <product_id>          - Delete a product using soft delete.
product_deleted <search>          - Show soft deleted products (or filter by <search>).
product_restore <product_id>      - Un-delete a product that was soft deleted.
```

<details>
<summary>Men jag har inte AUTO_INCREMENT som id på produkten?</summary>

Ok, du kanske valde att lägga till ett eget id på produkten? I så fall kan du anpassa kommandot så här:

```bash
product_add <title> <base_price> <id> - Add a new product with its base price and id.
```

</details>

Det räcker att man kan redigera produktens namn och pris vid `product_edit`. Tanken med "edit" är att man får upp en ny meny (eller liknande) där man kan redigera detaljer om produkten. En variant är att man kan klicka igenom alla värden som kan ändras och antingen skriver man in ett nytt värde för att ändra, eller så trycker man enter och då ändras inte värdet. Men exakt hur man löser det kan variera.

Använd "soft delete" när en produkt raderas med `product_del`.

När en produkt är "soft" raderad så skall den inte synas via kommandot `product <search>` (från kmom07).

Man skall kunna återställa en raderad produkt med hjälp av `product_restore`.

### Krav 5 (optionellt): Picklist & Ship

[kmom07] Det finns stöd för att skapa en order med orderrader och "beställa" sin order, varje orderrad har en produkt, antal och ett styckepris. I produktens tabell finns ett inköpspris som kan skilja sig från försäljningspriset.

[kmom08] Det finns stöd för att lägga in och ta bort produkter på lagrets hyllor. På varje lagerhylla ligger ett antal av produkten. En lagerhylla kan innehålla flera olika produkter.

Implementera följande menyval.

```bash
picklist <order_id>     - Present a picklist for the order.
ship <order_id>         - Ship the order.
```

I terminalklienten kan man skapa en plocklista för en order med `picklist <order_id>`. Det är en rapport som visar hur många av respektive beställd produkt finns i lagret (eller inte). Rapporten visar på vilken lagerhylla som produkten finns. Plocklistan är till den som skall paketera beställningen i en låda för att levereras till beställaren.

Man kan markera en order som "skickad" via `ship <order_id>`. Det betyder att ordern ändrar status till att vara "paketerad och ivägskickad till kund".

När man skickar en order så skall antalet produkter på lagret räknas ned. Förslagsvis så använder du rapporten i picklist för att veta vilka lagerhyllor som skall räknas ned. 

Håll det enkelt när du testar. Undvik att placera en produkt på flera lagerhyllor. Se till att det finns tillräckligt med produkter på lagret. Undvik specialfall. Undvik felfall. Håll det enkelt. Krångla inte till det.

I databasen skall det finnas minst följande data rörande ordrar.

1. Lägg in minst 5 ordrar där följande skall vara uppfyllt:
    1. En order har ingen orderrad.
    1. Fyra av ordrarna har en eller flera orderrader.
    1. En av ordrarna är markerad som ännu ej beställd.
    1. En av ordrarna är markerad som beställd.
    1. Två av ordrarna är markerade som skeppad/skickad.

### Krav 6 (optionellt): Invoice & Pay

Detta kravet förutsätter att du löst krav 5.

När en leverans skickas så genereras automatiskt ett underlag för en faktura med ett fakturanummer. Fakturan innehåller samma orderrader som ordern.

När man skriver ut en faktura kan man se detaljer om själva fakturan som visar både datum för när ordern skapades och när fakturan skapades samt vilken kund det gäller. Det skall finnas ett senaste betaldatum för fakturan som är 30 dagar från när fakturan skapades.

För varje orderrad, eller fakturarad, så ser man produktens id och namn samt antal, försäljningspris och totalen. Man ser också inköpspriset per styck (produktens baspris) och en summa som berättar hur mycket företaget tjänade på just denna fakturaraden.

I sista delen ser man totala beloppet som fakturan är på, det som kunden skall betala. Man ser också totalt hur mycket företaget tjänade på just denna fakturan.

Detta med företages vinst på försäljningen är ju inte den informationen som kunden får på sin faktura, denna faktura innehåller lite mer så att företagets controller kan undersöka hur mycket pengar man tjänar på varje faktura. Det är en faktura med "extra allt".

När fakturan skrivs ut så skall det tydligt framgå, i sista utskriften att fakturan är betald eller ej, man skall också se när fakturan betalades.

Implementera följande menyval.

```bash
invoices <search>      - Show a list of all invoices (or filter by search).
invoice <invoice_id>   - Show all details of the invoice.
paid <invoice_id>      - Mark the invoice as payed.
```

Det skall finnas minst två fakturor i databasen och av dessa skall en vara betald och den andra skall ännu inte vara betald.

För övrigt, håll det enkelt. Undvik specialfall. Krångla inte.
